## Overview

This endpoint allows administrators to create user accounts for internal staff members (employees) with specific roles. It is separate from the public registration endpoint and provides role assignment capabilities.

Key features:
- Create accounts for Administrador, Gerente, Supervisor, Vendedor, or Almacenista roles
- Prevents creation of Cliente (customer) accounts through this endpoint
- Validates email uniqueness across the system
- Requires password confirmation

## Authentication

<ParamField header="Authorization" type="string" required>
  Bearer token for authentication. User must have the **Administrador** role.
</ParamField>

<Warning>
  Only administrators can create employee accounts. This endpoint is not accessible to other roles.
</Warning>

## Request Body

<ParamField body="nombre" type="string" required>
  Employee's first name (max 100 characters)
</ParamField>

<ParamField body="apellido_paterno" type="string" required>
  Employee's paternal last name (max 100 characters)
</ParamField>

<ParamField body="apellido_materno" type="string">
  Employee's maternal last name (max 100 characters, optional)
</ParamField>

<ParamField body="correo" type="string" required>
  Employee's email address (must be unique in the system)
</ParamField>

<ParamField body="contrasena" type="string" required>
  Password for the account (minimum 8 characters)
</ParamField>

<ParamField body="contrasena_confirmation" type="string" required>
  Password confirmation (must match `contrasena`)
</ParamField>

<ParamField body="rol_id" type="number" required>
  Role ID to assign to the employee. Must be a valid role ID from the `roles` table.
  
  Common role IDs:
  - 1: Administrador
  - 2: Gerente
  - 4: Supervisor
  - 5: Vendedor
  - 6: Almacenista
  
  Note: Cliente role (typically ID 3) is not allowed via this endpoint.
</ParamField>

## Example Request

```bash
curl -X POST http://127.0.0.1:8000/api/admin/crear-empleado \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -H "Accept: application/json" \
  -d '{
    "nombre": "Carlos",
    "apellido_paterno": "Ramirez",
    "apellido_materno": "Lopez",
    "correo": "carlos.ramirez@solare.com",
    "contrasena": "SecurePass123!",
    "contrasena_confirmation": "SecurePass123!",
    "rol_id": 2
  }'
```

## Error Responses

### 400 Bad Request - Cliente Role

```json
{
  "message": "Para crear clientes use la ruta de registro pública"
}
```

Returned when attempting to create a Cliente account through this endpoint. Use the public `/api/register` endpoint instead.

### 401 Unauthorized

```json
{
  "message": "Unauthenticated."
}
```

### 403 Forbidden

```json
{
  "message": "No tiene permisos para acceder a este recurso."
}
```

Returned when the authenticated user does not have the Administrador role.

### 422 Unprocessable Entity - Validation Errors

```json
{
  "message": "The given data was invalid.",
  "errors": {
    "correo": [
      "The correo has already been taken."
    ],
    "contrasena": [
      "The contrasena confirmation does not match."
    ],
    "rol_id": [
      "The selected rol id is invalid."
    ]
  }
}
```

Returned when validation fails. Common validation errors:
- Email already exists in the system
- Password and confirmation don't match
- Password is less than 8 characters
- Invalid role ID
- Required fields are missing

## Implementation Notes

- Password is hashed using Laravel's `Hash::make()` before storage
- Email uniqueness is validated against the `usuarios` table
- The endpoint prevents creation of Cliente accounts (enforced at `app/Http/Controllers/Auth/AuthController.php:91`)
- No token is generated for the new employee - they must use `/api/login` to authenticate
- No Cliente profile record is created (unlike public registration)
- The response includes the full user object with eager-loaded role relationship
- Controller location: `app/Http/Controllers/Auth/AuthController.php:80`

## Workflow

1. Administrator authenticates and obtains access token
2. Administrator calls this endpoint with employee details and desired role
3. System validates input data:
   - Checks email uniqueness
   - Validates password strength and confirmation
   - Validates role ID exists
   - Prevents Cliente role assignment
4. User record is created in `usuarios` table with hashed password
5. Employee can now log in using `/api/login` with their credentials

## Security Considerations

- Only administrators can create employee accounts
- Passwords are hashed before storage
- Email uniqueness prevents duplicate accounts
- Password confirmation required to prevent typos
- Cliente accounts cannot be created through this endpoint
- No automatic token generation - forces proper login flow
