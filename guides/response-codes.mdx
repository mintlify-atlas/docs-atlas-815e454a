The SOLARE API uses standard HTTP status codes to indicate the success or failure of requests. This guide provides a comprehensive reference with real examples from the API.

## Status Code Categories

| Range | Category | Meaning |
|-------|----------|----------|
| 2xx | Success | Request was successful |
| 4xx | Client Error | Problem with the request |
| 5xx | Server Error | Problem on the server |

<Note>
  All responses include JSON bodies with additional context. Never rely solely on status codes.
</Note>

## Success Responses (2xx)

### 200 OK

**Meaning:** Request succeeded and returned data

**Used for:**
- GET requests that return data
- Successful authentication (login)
- Successful updates (PUT)

<CodeGroup>
```json GET Products (200 OK)
GET /api/productos

{
  "status": "success",
  "data": [
    {
      "id": 5,
      "nombre": "Sofá Moderno Capitoné",
      "precio_base": 25000.00,
      "categoria": {"nombre": "Salas"},
      "variantes": [...]
    }
  ]
}
```

```json Login Success (200 OK)
POST /api/login

{
  "message": "Inicio de sesión exitoso",
  "access_token": "eyJ0eXAiOiJKV1QiLCJhbGc...",
  "token_type": "Bearer",
  "user": {
    "id": 1,
    "nombre": "Admin",
    "correo": "admin@solare.com",
    "rol": "Administrador"
  }
}
```

```json Update Stock (200 OK)
PUT /api/inventario/12

{
  "status": "success",
  "message": "Inventario actualizado correctamente",
  "nuevo_stock": 108
}
```

```json Get Profile (200 OK)
GET /api/profile

{
  "id": 15,
  "nombre": "Juan",
  "apellido_paterno": "Pérez",
  "correo": "juan@example.com",
  "rol": {
    "id": 3,
    "nombre": "Cliente"
  }
}
```
</CodeGroup>

**Implementation Examples:**

```php
]);
```

## Client Errors (4xx)

### 400 Bad Request

**Meaning:** Invalid request or business logic violation

**Used for:**
- Business rule violations
- Invalid stock operations
- Attempting to create customers via admin endpoint

<CodeGroup>
```json Negative Stock (400)
PUT /api/inventario/12
Body: {"tipo": "salida", "cantidad": 1000}

Response:
{
  "status": "error",
  "message": "El stock no puede ser negativo."
}
```

```json Wrong Customer Creation Route (400)
POST /api/admin/crear-empleado
Body: {"rol_id": 3, ...}  // 3 = Cliente

Response:
{
  "message": "Para crear clientes use la ruta de registro pública"
}
```
</CodeGroup>

**Implementation Examples:**

```php
// app/Http/Controllers/Api/InventoryController.php:57
if ($nuevaCantidad < 0) {
    throw new \Exception("El stock no puede ser negativo.");
}

// app/Http/Controllers/Auth/AuthController.php:92
if ($request->rol_id == $rolCliente->id) {
    return response()->json([
        'message' => 'Para crear clientes use la ruta de registro pública'
    ], 400);
}
```

### 401 Unauthorized

**Meaning:** Authentication required but missing or invalid

**Used for:**
- Missing Authorization header
- Invalid/expired token
- Incorrect credentials

<CodeGroup>
```json Missing Token (401)
GET /api/profile
# Missing: Authorization header

Response:
{
  "message": "Unauthenticated."
}
```

```json Invalid Token (401)
GET /api/profile
Authorization: Bearer invalid_token_here

Response:
{
  "message": "Unauthenticated."
}
```

```json Wrong Credentials (422, not 401)
POST /api/login
Body: {"correo": "user@example.com", "contrasena": "wrong"}

Response:
{
  "message": "Las credenciales proporcionadas son incorrectas.",
  "errors": {
    "correo": [
      "Las credenciales proporcionadas son incorrectas."
    ]
  }
}
```
</CodeGroup>

**Implementation:**

```php
// Laravel Sanctum automatically handles 401 responses
// When token is missing or invalid
Route::middleware('auth:sanctum')->group(function () {
    // Protected routes
});

// app/Http/Controllers/Auth/AuthController.php:127
if (!$user || !Hash::check($request->contrasena, $user->contrasena)) {
    throw ValidationException::withMessages([
        'correo' => ['Las credenciales proporcionadas son incorrectas.'],
    ]);
}
```

<Warning>
  Note that incorrect login credentials return **422** (validation error) rather than 401, following Laravel convention.
</Warning>

### 403 Forbidden

**Meaning:** Authenticated but lacking required permissions

**Used for:**
- Insufficient role permissions
- Missing role assignment
- Non-customer trying to place order

<CodeGroup>
```json Role Not Allowed (403)
GET /api/admin/dashboard
Authorization: Bearer <customer_token>

Response:
{
  "message": "Acceso denegado: Tu rol (Cliente) no tiene permiso para esta acción."
}
```

```json No Role Assigned (403)
GET /api/inventario
Authorization: Bearer <token_without_role>

Response:
{
  "message": "Sin permisos de acceso"
}
```

```json Not a Customer (403)
POST /api/pedidos
Authorization: Bearer <admin_token>  // Admin without Cliente profile

Response:
{
  "message": "El usuario no está registrado como cliente"
}
```
</CodeGroup>

**Implementation:**

```php
// app/Http/Middleware/CheckRole.php:21
if (!$request->user() || !$request->user()->rol) {
    return response()->json(['message' => 'Sin permisos de acceso'], 403);
}

if (!in_array($request->user()->rol->nombre, $roles)) {
    return response()->json([
        'message' => 'Acceso denegado: Tu rol (' . 
                    $request->user()->rol->nombre . 
                    ') no tiene permiso para esta acción.'
    ], 403);
}

// app/Http/Controllers/Api/PedidoController.php:30
if (!$cliente) {
    return response()->json([
        'message' => 'El usuario no está registrado como cliente'
    ], 403);
}
```

**Difference from 401:**

| Status | Authentication | Authorization |
|--------|----------------|---------------|
| 401 | ❌ Not authenticated | N/A |
| 403 | ✅ Authenticated | ❌ Lacks permission |

**Used for:**
- Missing required fields
- Invalid data types
- Failed validation rules
- Duplicate email

```json Missing Fields (422)
      "The contrasena field is required."
    ],

      "The telefono field is required."
    ]
  }
```

```json Duplicate Email (422)
POST /api/register

Response:

  "message": "The given data was invalid.",
  "errors": {
    "correo": [
```

```json Password Mismatch (422)
  "message": "The given data was invalid.",
  "errors": {
```

```json Invalid Order (422)
POST /api/pedidos

  "productos": [
    {
      "variante_id": 9999,  // Doesn't exist

    }

  "errors": {

      "The selected productos.0.variante_id is invalid."

    ]
  }

```

```json Invalid Stock Type (422)

  "errors": {
    "tipo": [
      "The selected tipo is invalid."
    ]
  }
```

**Implementation:**

```php
// Laravel's automatic validation
// app/Http/Controllers/Auth/AuthController.php:22
$request->validate([
    'nombre' => 'required|string|max:100',
    'apellido_paterno' => 'required|string|max:100',
    'correo' => 'required|email|unique:usuarios,correo',
    'contrasena' => 'required|string|min:8|confirmed',
    'telefono' => 'required|string|max:20',
]);
// Automatically returns 422 if validation fails
```

<Note>
  422 responses always include an `errors` object with field-specific messages. Parse this to show user-friendly error messages.
</Note>

## Server Errors (5xx)

### 500 Internal Server Error

**Meaning:** Unexpected server-side error

**Used for:**
- Database errors
- Uncaught exceptions
- Transaction failures
- Configuration issues

<CodeGroup>
```json Order Processing Error (500)
POST /api/pedidos
# Database constraint violation or trigger failure

Response:
{
  "status": "error",
  "message": "Error al procesar el pedido: SQLSTATE[23000]: Integrity constraint violation"
}
```

```json Registration Error (500)
POST /api/register
# Database connection issue

Response:
{
  "message": "Error al registrar usuario",
  "error": "SQLSTATE[HY000]: General error: 1205 Lock wait timeout exceeded"
}
```

```json Production Error (500)
# When APP_DEBUG=false

Response:
{
  "message": "Server Error"
}
```
</CodeGroup>

**Implementation:**

```php
// app/Http/Controllers/Api/PedidoController.php:70
catch (\Exception $e) {
    DB::rollBack();
    return response()->json([
        'status' => 'error',
        'message' => 'Error al procesar el pedido: ' . $e->getMessage()
    ], 500);
}

// app/Http/Controllers/Auth/AuthController.php:61
catch (\Exception $e) {
    return response()->json([
        'message' => 'Error al registrar usuario',
        'error' => $e->getMessage()
    ], 500);
}
```

<Warning>
  **Production Security:** Set `APP_DEBUG=false` in production `.env` to hide detailed error messages and prevent information disclosure.
</Warning>

### Other 5xx Codes

While not explicitly implemented in SOLARE, these may be returned by infrastructure:

| Code | Name | When It Occurs |
|------|------|----------------|
| 502 | Bad Gateway | Reverse proxy can't reach Laravel |
| 503 | Service Unavailable | Server maintenance or overload |
| 504 | Gateway Timeout | Request took too long to process |

## Response Code Decision Tree

```
Is the request properly formed?
  ├─ No → 400 Bad Request
  └─ Yes → Is authentication provided?
      ├─ No → 401 Unauthorized
      └─ Yes → Is authentication valid?
          ├─ No → 401 Unauthorized
          └─ Yes → Does user have required role?
              ├─ No → 403 Forbidden
              └─ Yes → Does the resource exist?
                  ├─ No → 404 Not Found
                  └─ Yes → Is the data valid?
                      ├─ No → 422 Unprocessable Entity
                      └─ Yes → Did the operation succeed?
                          ├─ No → 500 Internal Server Error
                          └─ Yes → Is it a creation?
                              ├─ Yes → 201 Created
                              └─ No → 200 OK
```

## Client Implementation Guide

### Handling Status Codes

<CodeGroup>
```javascript JavaScript/Axios
import axios from 'axios';

const api = axios.create({
  baseURL: 'http://127.0.0.1:8000/api',
  headers: {
    'Accept': 'application/json',
    'Content-Type': 'application/json'
  }
});

// Response interceptor
api.interceptors.response.use(
  response => response.data,  // 2xx responses
  error => {
    const status = error.response?.status;
    const data = error.response?.data;

    switch (status) {
      case 401:
        // Redirect to login
        console.error('Authentication required');
        window.location.href = '/login';
        break;
      
      case 403:
        // Show permission error
        console.error('Permission denied:', data.message);
        break;
      
      case 422:
        // Show validation errors
        Object.keys(data.errors || {}).forEach(field => {
          console.error(`${field}:`, data.errors[field].join(', '));
        });
        break;
      
      case 404:
        console.error('Resource not found');
        break;
      
      case 500:
        console.error('Server error:', data.message);
        break;
      
      default:
        console.error('Unexpected error:', data.message);
    }

    return Promise.reject(error);
  }
);
```

```python Python
import requests
from typing import Dict, Any

class SolareAPI:
    BASE_URL = 'http://127.0.0.1:8000/api'
    
    def __init__(self, token: str = None):
        self.session = requests.Session()
        self.session.headers.update({
            'Accept': 'application/json',
            'Content-Type': 'application/json'
        })
        if token:
            self.session.headers['Authorization'] = f'Bearer {token}'
    
    def _handle_response(self, response: requests.Response) -> Dict[Any, Any]:
        status = response.status_code
        
        try:
            data = response.json()
        except ValueError:
            data = {'message': 'Invalid JSON response'}
        
        if status == 200 or status == 201:
            return data
        elif status == 401:
            raise AuthenticationError('Authentication required')
        elif status == 403:
            raise PermissionError(data.get('message', 'Forbidden'))
        elif status == 404:
            raise NotFoundError('Resource not found')
        elif status == 422:
            raise ValidationError(data.get('errors', {}))
        elif status == 500:
            raise ServerError(data.get('message', 'Server error'))
        else:
            raise APIError(f"Unexpected status {status}: {data.get('message')}")
    
    def get(self, endpoint: str) -> Dict[Any, Any]:
        response = self.session.get(f'{self.BASE_URL}{endpoint}')
        return self._handle_response(response)
    
    def post(self, endpoint: str, data: Dict) -> Dict[Any, Any]:
        response = self.session.post(f'{self.BASE_URL}{endpoint}', json=data)
        return self._handle_response(response)

# Custom exceptions
class APIError(Exception): pass
class AuthenticationError(APIError): pass
class ValidationError(APIError): pass
class NotFoundError(APIError): pass
class ServerError(APIError): pass
```
</CodeGroup>

### Status Code Testing

Postman test examples:

```javascript
// Test successful response
pm.test("Status is 200 OK", function () {
    pm.response.to.have.status(200);
});

pm.test("Resource created (201)", function () {

    pm.response.to.have.status(403);

// Test validation

    pm.response.to.have.status(422);
    pm.expect(pm.response.json()).to.have.property('errors');
});
```

## Best Practices

<AccordionGroup>
  <Accordion title="Always Check Status Code First">
    Before parsing the response body, check the HTTP status code to determine the response category (success, client error, server error).
    
    ```javascript
    if (response.status >= 200 && response.status < 300) {
      // Success
    } else if (response.status >= 400 && response.status < 500) {
      // Client error
    } else if (response.status >= 500) {
      // Server error
    }
    ```
  </Accordion>
  
  <Accordion title="Parse Error Responses">
    All error responses include JSON bodies with details:
    
    - 422: Check `errors` object for field-specific messages
    - 403: Check `message` for role information
    - 500: Log full error details for debugging
  </Accordion>
  
  <Accordion title="Implement Retry Logic">
    Only retry on specific status codes:
    
    - **Retry:** 500, 502, 503, 504 (after delay)
    - **Don't retry:** 400, 401, 403, 404, 422
  </Accordion>
  
  <Accordion title="Handle Token Expiration">
    401 responses may indicate expired tokens. Implement refresh logic or redirect to login:
    
    ```javascript
    if (status === 401) {
      // Clear stored token
      localStorage.removeItem('token');
      // Redirect to login
      window.location.href = '/login';
    }
    ```
  </Accordion>
</AccordionGroup>

## Summary Table

| Code | Name | Retry? | User Action | Technical Action |
|------|------|--------|-------------|------------------|
| 200 | OK | No | None | Process data |
| 201 | Created | No | Show success | Store resource ID |
| 400 | Bad Request | No | Fix input | Check business rules |
| 401 | Unauthorized | No | Login | Provide valid token |
| 403 | Forbidden | No | Contact admin | Use correct role |
| 404 | Not Found | No | Check URL | Verify resource exists |
| 422 | Validation Error | No | Fix fields | Show error messages |
| 500 | Server Error | Yes | Try again | Report to support |
